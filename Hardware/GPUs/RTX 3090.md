# RTX 3090

`RTX 3090` is the most common "affordable VRAM play" in the local-LLM stack: 24GB of memory on a single consumer GPU at a price often far below newer 24GB consumer alternatives. It is old enough that raw compute is behind `RTX 4090`, but it remains a strong value point when the question is **can it run this model**, not **how fast can it finish every benchmark**.

---

## 🎯 Why it matters for LLMs
The 24GB VRAM envelope is the hard stop for single-card workloads. In practice, that is enough to push many 30–34B models in quantized form and run smaller models comfortably.

[[LLM Under Your Floorboards]] treats this card as the budget-tier entry for serious local work:
- `$800-1,000` used in the quick reference table
- best for 13B, 34B Q4, and similar models depending on context window
- much slower than a `4090`, but often much cheaper per usable VRAM

It is also useful as a stepping stone into:
- `llama.cpp` / `Ollama` on a single card
- multi-GPU inference where bridge-level `NVLink` is acceptable
- local quantized workflows where VRAM budget is the first-order constraint

## 🧮 Core specs (practical)

| Metric | RTX 3090 (24GB) | Why this matters for local LLMs |
|--------|------------------|----------------------------------|
| VRAM | 24GB | Sets model size ceiling on single GPU |
| Memory bandwidth | 936 GB/s | Directly impacts generation throughput |
| Power | ~350W | Affects PSU and thermal design |
| FP16 compute | ~35.6 TFLOPS | Slower than newer cards, important for max t/s |
| Consumer NVLink bridge | 2-GPU only | Enables 2x3090 scaling experiments |
| GDDR6X | yes | Higher memory efficiency than many prior gen cards |

The practical rule still holds: for local inference, **VRAM + bandwidth typically dominate over raw flops** more than in image workloads.

## ⚙️ CUDA + driver support (for inference + PufferLib workflows)

`RTX 3090` is based on compute capability `8.6`, so it is supported by CUDA stacks from Ampere-era releases onward.

- `CUDA Toolkit 11.1` was the first mainstream Ampere-compatible line for `sm_86`.
- NVIDIA minor-version compatibility currently gives these practical minimums:
  - `CUDA 11.x` needs minimum driver `>= 450` (Linux `>= 450.80.02`, Windows `>= 452.39` in NVIDIA docs).
  - `CUDA 12.x` needs minimum driver `>= 525`.
  - `CUDA 13.x` needs minimum driver `>= 580`.
- In 2026-era local stacks, practical setups are usually:
  - `CUDA 12.x` runtime + corresponding modern driver, or
  - `CUDA 13.x` runtime + corresponding modern driver.
- In practice for local ML stacks, keep toolchain and framework versions aligned:
  - Python env: `torch`, `jax`, `transformers`, and any backend built for `sm_86`.
  - Mismatch risk is almost always from framework wheel/driver mismatch, not from 3090 hardware limits.

For PufferLib specifically:
- `PufferLib` itself does not add a separate GPU architecture requirement.
- Its compatibility depends on your selected backend (`PyTorch` or `JAX`) and environment loop being stable.
- For RL workflows, this card is generally sufficient for serious experimentation in policies, value models, and vectorized rollouts; bottlenecks usually become:
  - environment step cost,
  - `batch_size`,
  - and PPO/AC rollout memory layout.

Useful official checks before install:
- NVIDIA product page confirms `CUDA Capability: 8.6`.
- NVIDIA minor-version compatibility table gives the minimum driver branches per CUDA major release.
- `pytorch.org` install selector for exact CUDA wheel compatibility.

## 🧠 Model fit reality check
Use the sizing logic in [[LLM Inference Hardware]] as the base reference and convert it into practical 3090 decisions:

| Model target | Typical path on single `RTX 3090` | Practical notes |
|--------------|-----------------------------------|-----------------|
| 7-8B | `Q4` / `Q5` full-fit | Good experience |
| 13B | `Q4` typically works | depends on context |
| 30-34B | `Q4` often works with tuning | close to VRAM edge on long contexts |
| 70B | usually `Q2-Q3` + aggressive settings only | typically not ideal on single card |
| 405B | not practical on single card | only possible through serious distributed inference |

### What this means for PufferLib vs LLM goals

- `PufferLib` workloads:
  - policy/value networks and PPO/AC variants usually fit well on 24GB.
  - most training value is in environment design + rollout throughput, not raw model size.
- `LLM under your floorboards` workflows:
  - strong budget baseline for 30B-class quantized models,
  - viable for 70B only in very compressed forms and with context tradeoffs,
  - 2x3090 + NVLink helps only modestly for larger deployments due to overhead.

When context grows, KV cache memory becomes a real tax. A larger context can push even previously comfortable sizes into slow spill or OOM territory.

## 🔗 NVLink: what it does and what it does not do
`RTX 3090` cards can participate in a consumer `NVLink` bridge, but this is not magical shared memory in the way many people assume.

### What NVLink does help
1. Reduces inter-GPU latency compared with PCIe.
2. Improves tensor/pipeline parallel scaling for supported inference backends.
3. Allows larger effective model layouts than 24GB alone.

### What NVLink does *not* do
1. It does not make 48GB appear as one flat 48GB VRAM pool in every stack.
2. It does not remove context replication costs.
3. It is still limited to 2 GPUs for this card family.

The existing note-level shorthand in the ecosystem (`2x 3090 = 48GB`) is useful only as an *effective fit estimate*. In practice, actual usable memory depends on engine partition strategy and replication overhead.

## 🧪 Can two 3090s behave like a 48GB system?
For some inference stacks, yes, with caveats:

| Setup | Effective capacity | Performance expectation |
|-------|--------------------|------------------------|
| 1x RTX 3090 | 24GB | Solid on 30-34B quantized targets |
| 2x RTX 3090 + `NVLink` | roughly model-sharded up to ~48GB | Better model coverage, slower than true single-socket VRAM bandwidth |
| 2x RTX 3090 over PCIe only | uneven/limited scaling | often materially slower than bridgeed version |

What this means for [[LLM Under Your Floorboards]]:
- If budget is capped, 2x `RTX 3090` can be a middle ground for 34B workflows.
- If you need stable 70B quality, a modern 48GB workstation card or datacenter setup is often more efficient.

In local benchmarks in this vault, consumer bridge setups are still significantly below datacenter multi-GPU bandwidth profiles.

## 💸 Pricing and buying guidance
From the current local-LLM notes:
- Budget row in [[LLM Under Your Floorboards]] shows used 3090 around `$800-1,000`.
- `LLM Inference Hardware` values show used single-card inference around `$700`.
- For throughput economics, the 3090 is frequently listed as a good-value entry card for 8B workloads.

A useful buying rule:
- spend on the card first, then add RAM and NVMe before chasing extra GPU upgrades;
- buy at least 64GB system RAM for comfortable model loading plus cache headroom;
- verify bridge support and BIOS/Passthrough behavior if multi-GPU is planned.

For power/cooling planning, pair with:
- good 750W+ PSU for dual-card rigs (or more for overclock headroom),
- adequate airflow and noise-aware chassis because the card is 350W class.

## 🧩 How to configure for local inference

### Single card (single-GPU)
- start with a quantized 4-bit `GGUF` workflow
- reduce context length until warm stability is reached
- lock `gpu-memory-utilization` near `0.90` unless running many parallel clients

### Two cards (`RTX 3090` + bridge)
- expose both cards and force tensor-parallel inference only with frameworks that support it
- keep monitor-level checks for cross-card stutter and uneven memory use
- expect diminishing returns above 2 cards because this family does not support wide consumer mesh
