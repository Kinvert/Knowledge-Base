# RTX 4090

`RTX 4090` is usually the most practical single-card consumer choice for serious local LLM inference in 2026. It keeps the same 24GB VRAM class as the `RTX 3090`, but with much higher throughput than older gen cards.

For practical local workflow, this card is the “mainline” consumer choice when you want a balance of:
- model size capability from 8B to 34B quantized workloads,
- stronger generation speed than 3090-class cards,
- and relatively broad ecosystem support.

---

## 🎯 Why it matters for local LLM

In local-LLM terms, the `RTX 4090` solves a specific issue:

- You get enough VRAM to run many 30-34B quantized models comfortably.
- It performs much better than previous generation 24GB cards, especially at larger context windows.
- For budget VRAM scaling beyond 24GB, it remains a strong base card for 2-way bridgeed pairs.

The 3090 and 4090 both sit at 24GB, but the 4090’s inference throughput and memory subsystem make it a much better default if you can afford it.

## 🧠 Model fit guide

Using the size mapping from local inference tables, this is a practical fit guide:

| Target model class | Typical fit mode | Result |
|--------------------|------------------|--------|
| 7-8B | single card, moderate quantization | easy |
| 13B | single card, Q4/Q5 | comfortable |
| 30-34B | single card, Q4 | often the sweet spot |
| 70B | two 4090 (bridge) or quantized single-card setup | possible but quality/speed tradeoff |

The largest difference versus older cards is not “one more VRAM” (still 24GB), it is throughput and execution stability in long contexts.

---

## 🔗 NVLink and two-card behavior

`RTX 4090` cards support the consumer `NVLink` bridge path, which supports **2-card pairs only** in this consumer segment.

### What it enables
1. Lower inter-GPU transfer cost than PCIe for compatible parallel setups.
2. Better practical scaling than dual-PCIe in tensor-parallel inference setups.
3. Higher combined effective capacity for multi-GPU model partitioning.

### What it does not eliminate
1. It does not convert two cards into a true single 48GB flat memory pool in every workload.
2. It does not remove context replication overhead.
3. It does not support more than two-way consumer bridge topologies.

Short version:
- Yes, 2x4090 can handle larger targets than a single card.
- No, you should not assume it behaves like a single 48GB card across all stacks.

## 💸 Pricing and value today

In existing local inference references, this card is typically described as:
- `~$1600-2000` for a strong used/buying range for planning
- a major speed step versus 24GB older cards

Interpret this as a **moving range**:
- cards on older used market listings move fast in price,
- some “new old stock” can sit above this when supply shifts,
- local performance gains matter as much as purchase price for cost/performance.

If you are comparing against `A100` used builds, the 4090 still wins for single-device developer workflows and local dev loops because of lower power complexity and easier setup.

## 🧪 Deep dive vs adjacent cards

| Option | VRAM | Throughput style | Best role |
|--------|------|------------------|-----------|
| RTX 3090 | 24GB | slower baseline | budget entry |
| RTX 4090 | 24GB | strongest consumer baseline | best balanced 30-34B single-card workflow |
| RTX 5090 | 32GB | much faster + more memory headroom | low latency on larger models |
| RTX A6000 | 48GB | workstation VRAM-heavy | less efficient than 4090 per watt, but higher memory ceiling |
| 2x RTX 4090 + NVLink | 48GB effective split | strong budget multi-card path | 70B workflows before full datacenter tier |

One key result from this comparison:
- If you need consistent 70B quality, 2x4090 is a transition option.
- If you need simpler setup, a single 4090 is still clean for most 34B-class usage.

---

## 🧩 How to tune local inference with RTX 4090

### Single card
- Start with `Q4_K_M` style quantization where available.
- Reduce context if you hit memory spikes from KV cache growth.
- Keep VRAM utilization under 90% when serving multiple users.

### Two cards (if bridge path is used)
- Prefer inference stacks with explicit tensor parallel controls.
- Benchmark before and after adding bridge; gains vary by kernel/backend.
- If overhead is high, stay conservative on context and batch settings.
