# C99

**C99** is the 1999 ISO C standard, formally **ISO/IEC 9899:1999**. It was the first major update after ANSI C / C89 / C90 and introduced many features that people now think of as normal "modern C."

C99 is still common in embedded systems, portable C libraries, Linux-adjacent code, numerical code, and projects that want modern syntax without depending on C11 atomics or threading support.

---

## Quick Summary

C99 added:

- Mixed declarations and code.
- `//` comments.
- `long long`.
- `_Bool` and `stdbool.h`.
- Fixed-width integers in `stdint.h`.
- Portable integer format macros in `inttypes.h`.
- Designated initializers.
- Compound literals.
- Variable length arrays.
- Flexible array members.
- `restrict`.
- Standard `inline`.
- Variadic macros.
- `snprintf`.
- Complex numbers.
- Hexadecimal floating constants.
- `__func__`.

## Compiler Flags

Strict C99:

```bash
gcc -std=c99 -Wall -Wextra -Wpedantic file.c
clang -std=c99 -Wall -Wextra -Wpedantic file.c
```

GNU C99 dialect:

```bash
gcc -std=gnu99 file.c
clang -std=gnu99 file.c
```

Use `-std=c99` when checking portability. Use `-std=gnu99` when you intentionally depend on GNU extensions.

## Why C99 Mattered

C89/C90 forced a more old-school style:

```c
void f(void) {
    int x;
    int y;

    x = compute();
    y = x + 1;
}
```

C99 allows declarations near use:

```c
void f(void) {
    int x = compute();
    int y = x + 1;
}
```

That one change alone made C code easier to maintain. The rest of C99 added better integer types, better initialization syntax, safer formatting, and more expressive low-level patterns.

## Core Language Features

### Mixed Declarations and Code

Declare variables close to where they are used:

```c
int parse(const char *s) {
    int status = validate(s);
    if (status != 0) {
        return status;
    }

    int value = convert(s);
    return value;
}
```

### Single-Line Comments

```c
// This is standard in C99.
```

### `long long`

C99 standardizes `long long` and `unsigned long long`.

```c
long long x = 9223372036854775807LL;
```

### Boolean Type

```c
#include <stdbool.h>

bool ok = true;
```

`stdbool.h` maps `bool`, `true`, and `false` to the underlying C type `_Bool`.

### Designated Initializers

```c
struct Point {
    int x;
    int y;
};

struct Point p = {
    .x = 10,
    .y = 20,
};
```

Useful because the initializer does not silently depend on field order.

Array designators:

```c
int table[8] = {
    [0] = 1,
    [7] = 99,
};
```

### Compound Literals

```c
struct Point p = (struct Point){.x = 1, .y = 2};
```

Common for temporary structs:

```c
draw_point((struct Point){.x = 10, .y = 20});
```

### Flexible Array Members

```c
struct Packet {
    size_t len;
    unsigned char data[];
};
```

Allocation pattern:

```c
struct Packet *p = malloc(sizeof *p + payload_len);
p->len = payload_len;
```

This is the standard replacement for old `data[1]` trailing-array hacks.

### Variable Length Arrays

```c
void f(size_t n) {
    int scratch[n];
    /* use scratch */
}
```

VLAs are standard in C99. They are useful but controversial because allocation failure is hard to handle and large runtime stack allocations can be dangerous.

For portable or safety-critical code, prefer explicit allocation or fixed maximum buffers.

### `restrict`

`restrict` tells the compiler that, for the lifetime of the pointer, the object is only accessed through that pointer or values derived from it.

```c
void add(size_t n,
         float * restrict out,
         const float * restrict a,
         const float * restrict b) {
    for (size_t i = 0; i < n; i++) {
        out[i] = a[i] + b[i];
    }
}
```

This can improve vectorization and optimization, but it is a real contract. Violating it is undefined behavior.

### `inline`

```c
static inline int min_int(int a, int b) {
    return a < b ? a : b;
}
```

For helper functions in headers, `static inline` is usually the least surprising pattern.

## Library Features

### `stdint.h`

Fixed-width and minimum-width integer types:

```c
#include <stdint.h>

uint8_t byte;
uint32_t word;
int64_t signed_counter;
uintptr_t pointer_bits;
```

Important types:

| Type | Meaning |
|---|---|
| `int8_t`, `uint8_t` | Exactly 8-bit integer, if available |
| `int16_t`, `uint16_t` | Exactly 16-bit integer, if available |
| `int32_t`, `uint32_t` | Exactly 32-bit integer, if available |
| `int64_t`, `uint64_t` | Exactly 64-bit integer, if available |
| `intptr_t`, `uintptr_t` | Integer type capable of holding a pointer, if available |
| `size_t` | Object size type, from standard headers |

### `inttypes.h`

Portable format macros:

```c
#include <inttypes.h>
#include <stdint.h>
#include <stdio.h>

uint64_t x = 123;
printf("%" PRIu64 "\n", x);
```

### `snprintf`

```c
char buf[64];
int n = snprintf(buf, sizeof buf, "value=%d", 42);
```

Unlike `sprintf`, `snprintf` knows the buffer size.

### Variadic Macros

```c
#define LOG(fmt, ...) fprintf(stderr, "%s: " fmt "\n", __func__, __VA_ARGS__)
```

### `__func__`

```c
void f(void) {
    puts(__func__);
}
```

Expands to the current function name.

### Complex Numbers

```c
#include <complex.h>

double complex z = 1.0 + 2.0 * I;
```

Mostly relevant in numerical and scientific code.

## Example C99 Program

```c
#include <stdbool.h>
#include <stdint.h>
#include <stdio.h>

struct Point {
    int x;
    int y;
};

static bool in_bounds(struct Point p, int width, int height) {
    return p.x >= 0 && p.x < width && p.y >= 0 && p.y < height;
}

int main(void) {
    struct Point p = {.x = 3, .y = 4};
    uint32_t flags = 0;

    printf("point=(%d,%d), flags=%u\n", p.x, p.y, flags);
    return in_bounds(p, 10, 10) ? 0 : 1;
}
```

## C99 Pitfalls

### VLAs Can Blow the Stack

```c
void f(size_t n) {
    char buf[n];  // risky if n is large or attacker-controlled
}
```

Prefer:

```c
char *buf = malloc(n);
if (!buf) {
    return;
}
free(buf);
```

### `restrict` Is Easy to Misuse

This is wrong if `a` and `out` overlap:

```c
void scale(size_t n, float * restrict out, float * restrict a);
```

Only use `restrict` when the aliasing contract is true.

### `inline` Linkage Can Surprise You

For functions in headers, use:

```c
static inline int helper(int x) {
    return x + 1;
}
```

### C99 Has No Standard Atomics

C99 does not define portable atomic operations or a thread memory model. Use platform APIs, compiler intrinsics, locks, or move to [[C11]] / [[C17]] for standard atomics.

### Microsoft C Support Was Historically Late

Old MSVC versions had incomplete C99 support. Modern MSVC supports more C11/C17-era C, but portable cross-platform C code should still test compiler behavior instead of assuming full C99 coverage.

---

## When To Use C99

Use C99 when:

- You need wide embedded/toolchain compatibility.
- You want modern C syntax but not C11 atomics or `threads.h`.
- You are writing portable libraries with conservative compiler requirements.
- The project or coding standard already targets C99.
- You are working in environments where C11 support is partial or vendor-specific.

Avoid depending on full C99 when:

- The compiler is old or unusual.
- VLAs are forbidden by the project.
- The codebase must compile as C++ too.
- You need defined multithreaded memory behavior.
