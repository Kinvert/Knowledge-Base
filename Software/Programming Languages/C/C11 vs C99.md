# C11 vs C99

**C99** and **C11** are ISO C language standards. C99 was the big modernization step after C89/C90. C11 is a smaller but important update that added a memory model, atomics, optional threading support, Unicode-related syntax, static assertions, and several cleanup features.

Short version:

- **C99** is the baseline for "modern C" syntax: mixed declarations, `//` comments, VLAs, designated initializers, compound literals, `restrict`, `_Bool`, `stdint.h`, variadic macros, and `inline`.
- **C11** keeps most C99 features, adds concurrency-aware language/library pieces, and makes some C99 features optional, most notably variable length arrays.

---

## Timeline

| Standard | ISO Name | Common Compiler Flag | Main Character |
|---|---|---|---|
| C89 / C90 | ISO/IEC 9899:1990 | `-std=c89`, `-std=c90` | Classic ANSI C |
| C99 | ISO/IEC 9899:1999 | `-std=c99` | Big language/library modernization |
| C11 | ISO/IEC 9899:2011 | `-std=c11` | Atomics, memory model, safer compile-time checks |
| C17 / C18 | ISO/IEC 9899:2018 | `-std=c17`, `-std=c18` | Bug-fix revision of C11 |
| C23 | ISO/IEC 9899:2024-ish naming in practice | `-std=c23`, `-std=c2x` | Bigger modernization after C11/C17 |

For portable code today, **C11 or C17** is usually the conservative target. For embedded or old toolchains, **C99** is still common.

---

## Compiler Flags

GCC / Clang:

```bash
gcc -std=c99 -Wall -Wextra file.c
gcc -std=c11 -Wall -Wextra file.c
gcc -std=c17 -Wall -Wextra file.c
```

GNU dialects:

```bash
gcc -std=gnu99 file.c
gcc -std=gnu11 file.c
```

`gnu99` / `gnu11` mean "C99/C11 plus GNU extensions." Use strict `c99` or `c11` when checking portability.

## Major C99 Features

C99 added many features people now think of as normal C:

| Feature | Example | Why It Matters |
|---|---|---|
| Mixed declarations and code | `int x = f(); int y = x + 1;` inside blocks | No need to declare everything at block top |
| `//` comments | `// comment` | Common single-line comments |
| Designated initializers | `.x = 1, .y = 2` | Robust struct/array initialization |
| Compound literals | `(struct Point){1, 2}` | Temporary aggregate values |
| Variable length arrays | `int a[n];` | Stack arrays with runtime size |
| Flexible array members | `struct S { size_t n; int data[]; };` | Common trailing-buffer pattern |
| `restrict` | `int * restrict p` | Alias optimization contract |
| `_Bool` / `stdbool.h` | `bool ok = true;` | Standard boolean type |
| `inline` | `inline int f(void)` | Standard inline syntax |
| `long long` | `long long x;` | At least 64-bit integer type |
| `stdint.h` | `uint32_t`, `int64_t` | Fixed-width integer types |
| `inttypes.h` | `PRIu64` | Portable printf/scanf macros |
| Variadic macros | `#define log(fmt, ...)` | Macro wrappers around printf-like APIs |
| `snprintf` | `snprintf(buf, n, ...)` | Safer formatted strings |
| Complex numbers | `_Complex double` | Standard complex arithmetic |
| Hex floats | `0x1.8p+2` | Exact floating constants |
| `__func__` | current function name | Useful for logging/debugging |

Example:

```c
#include <stdbool.h>
#include <stdint.h>

struct Point {
    int x;
    int y;
};

bool in_bounds(struct Point p, int width, int height) {
    return p.x >= 0 && p.x < width && p.y >= 0 && p.y < height;
}

int main(void) {
    struct Point p = {.x = 3, .y = 4};
    uint32_t flags = 0;
    return in_bounds(p, 10, 10) ? (int)flags : 1;
}
```

## Major C11 Features

C11 did not replace C99 style. It mostly adds tools for safer, more explicit low-level programming.

| Feature | Header / Syntax | Why It Matters |
|---|---|---|
| Memory model | language-level | Defines data races and cross-thread visibility |
| Atomics | `stdatomic.h`, `_Atomic` | Portable lock-free-ish atomic operations |
| Threads | `threads.h` | Optional standard threading API |
| Static assertions | `_Static_assert` | Compile-time checks |
| Generic selection | `_Generic` | Type-based macro dispatch |
| Alignment | `stdalign.h`, `_Alignas`, `_Alignof` | Explicit alignment control/query |
| No-return functions | `stdnoreturn.h`, `_Noreturn` | Tell compiler a function never returns |
| Anonymous structs/unions | syntax support | Cleaner embedded/low-level layout types |
| Unicode literals | `u8`, `u`, `U` prefixes | UTF-8/UTF-16/UTF-32 literals |
| Bounds-checking interfaces | Annex K | Optional and rarely portable |
| Feature test macros | `__STDC_NO_*__` | Detect missing optional C11 pieces |

Example static assertion:

```c
#include <stdint.h>

_Static_assert(sizeof(uint32_t) == 4, "uint32_t must be 4 bytes");
```

Example `_Generic`:

```c
#include <stdio.h>

#define type_name(x) _Generic((x), \
    int: "int", \
    float: "float", \
    double: "double", \
    default: "other")

int main(void) {
    printf("%s\n", type_name(3.14));
}
```

Example atomics:

```c
#include <stdatomic.h>

atomic_int counter = 0;

void tick(void) {
    atomic_fetch_add_explicit(&counter, 1, memory_order_relaxed);
}
```

## C11 Optional Features

One subtle compatibility issue: C11 made some features optional. Implementations can define macros saying they do **not** support them.

| Macro | Means Missing |
|---|---|
| `__STDC_NO_ATOMICS__` | No `stdatomic.h` |
| `__STDC_NO_THREADS__` | No `threads.h` |
| `__STDC_NO_VLA__` | No variable length arrays |
| `__STDC_NO_COMPLEX__` | No complex arithmetic |

This means **C11 is not simply "C99 plus everything."** A C11 compiler may reject VLAs even though C99 required them.

Check optional support:

```c
#ifdef __STDC_NO_VLA__
#error "This code requires variable length arrays"
#endif
```

## VLA Difference

Variable length arrays are one of the biggest practical C99 vs C11 differences.

C99:

```c
void f(size_t n) {
    int tmp[n];
}
```

In C99, VLAs are a standard feature.

C11:

```c
#ifdef __STDC_NO_VLA__
/* VLAs unavailable */
#endif
```

In C11, VLAs are optional. Many compilers still support them, but relying on them can hurt portability, especially in embedded, kernel, Windows/MSVC, and safety-critical code.

Common replacement:

```c
int *tmp = malloc(n * sizeof *tmp);
if (!tmp) {
    return;
}

/* use tmp */

free(tmp);
```

Or use fixed-size buffers when the maximum is known:

```c
enum { MAX_ITEMS = 256 };
int tmp[MAX_ITEMS];
```

## Atomics and Memory Model

C99 has no standard thread memory model. You could use pthreads, compiler intrinsics, or volatile-heavy code, but the language itself did not define data races or portable atomic operations.

C11 defines:

- What a **data race** is.
- How atomic operations synchronize.
- Memory ordering modes such as `relaxed`, `acquire`, `release`, `acq_rel`, and `seq_cst`.
- Atomic types via `_Atomic` and `stdatomic.h`.

Important: `volatile` is **not** a thread synchronization primitive. Use atomics, locks, or platform primitives.

Typical C11 atomic flag:

```c
#include <stdatomic.h>
#include <stdbool.h>

atomic_bool stop_requested = false;

void request_stop(void) {
    atomic_store_explicit(&stop_requested, true, memory_order_release);
}

bool should_stop(void) {
    return atomic_load_explicit(&stop_requested, memory_order_acquire);
}
```

## Threads

C11 added `threads.h`, but support is uneven. POSIX systems often still use [[Pthreads]], and many portable codebases wrap threading behind their own abstraction.

C11 thread example:

```c
#include <threads.h>
#include <stdio.h>

int worker(void *arg) {
    puts((const char *)arg);
    return 0;
}

int main(void) {
    thrd_t t;
    thrd_create(&t, worker, "hello");
    thrd_join(t, NULL);
}
```

Practical note:

- `stdatomic.h` is more important than `threads.h`.
- Use `threads.h` only if your target compilers/libcs support it.
- For Unix-heavy code, pthreads remain more common.

## `inline` Gotcha

C99 standardized `inline`, but its linkage rules differ from GNU89 inline behavior. This can cause duplicate symbol or missing symbol errors when changing compiler modes.

Common safe pattern for header-only small functions:

```c
static inline int add1(int x) {
    return x + 1;
}
```

`static inline` avoids most C99/GNU inline linkage confusion for helper functions in headers.

## Migration: C99 to C11

Usually easy:

1. Compile with `-std=c11 -Wall -Wextra -Wpedantic`.
2. Check for VLA usage if portability matters.
3. Replace ad hoc atomic/thread hacks with `stdatomic.h` where available.
4. Add `_Static_assert` for ABI/layout assumptions.
5. Watch for library availability, especially `threads.h` and Annex K functions.

Good C11 additions to adopt even in conservative code:

```c
_Static_assert(sizeof(void *) == 8, "64-bit target required");
```

```c
#include <stdalign.h>

struct alignas(64) CacheLineCounter {
    int value;
};
```

```c
#include <stdatomic.h>

atomic_uint refcount;
```

## Migration: C11 to C99

Going backward is harder if you used C11-specific features.

| C11 Feature | C99 Replacement |
|---|---|
| `_Static_assert` | Macro trick or build-system check |
| `_Atomic` / `stdatomic.h` | Compiler intrinsics, pthread mutexes, platform atomics |
| `threads.h` | pthreads / Win32 threads / abstraction layer |
| `_Generic` | Type-specific function names or macros |
| `_Alignas` / `_Alignof` | Compiler extensions or manual padding |
| `_Noreturn` | Compiler attributes like `__attribute__((noreturn))` |
| Unicode literal prefixes | Avoid or use compiler-specific support |

If targeting old embedded compilers, C99 plus carefully gated compiler extensions is often more realistic than full C11.

---

## Practical Recommendation

| Situation | Pick |
|---|---|
| New portable systems code | C11 or C17 |
| Conservative embedded project | C99 |
| Code needs standard atomics | C11 |
| Code must support old MSVC or embedded C compilers | C99-ish subset |
| Safety-critical / MISRA-heavy code | Usually restricted C99/C11 subset |
| Want "modern C" without threads/atomics | C99 is enough |
| Want defined concurrent behavior | C11 or newer |

For many projects, the sweet spot is:

```bash
-std=c11 -Wall -Wextra -Wpedantic
```

or:

```bash
-std=c17 -Wall -Wextra -Wpedantic
```

C17 is effectively a corrected C11, not a feature-heavy new language.

| Category | C99 | C11 |
|---|---|---|
| Mixed declarations/code | Yes | Yes |
